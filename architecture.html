V2.0 Core Architecture

Sprinter Orchestration Engine

A durable, local-first workflow engine designed for complex, multi-step automation of Jira and GitHub operations.

Architectural Flow

Sprinter has evolved from a simple webhook worker into a fully decoupled, event-driven orchestration system. This architecture ensures high durability, observability, and concurrency control.

The Orchestrator Store acts as the single source of truth, using atomic file operations to move events and commands between lifecycle directories (pending → running → completed/failed).

Package Reference

Core

orchestrator

The central engine of the system. It manages the lifecycle of events and workflows.

  • models.py: Data structures for Events, Commands, and State.
  • engine.py: The state machine logic.
  • store.py: Durable filesystem persistence.
  • dispatcher.py: Process pool management.
Interface

webhooks

Lightweight HTTP server that receives external triggers and feeds them to the engine.

  • Jira webhook parsing and verification.
  • Event deduplication via FilesystemWebhookStore.
  • Legacy mode support for direct execution.
Interface

OrchestratorMCP

Codex-facing interface providing full visibility and manual control over automation.

  • Tools for workflow inspection (list, get, history).
  • Manual control (retry, pause, resume).
  • Worker pool monitoring.
  • Direct worker command enqueueing and dispatch.
  • Dedicated tools for export, planner, executor, PR, and reviewer workers.
Execution

workers

Standalone CLI tools that perform specific tasks. They are isolated by process.

  • export_jira_worker.py: Full data extraction.
  • planner_worker.py: Codex-based technical planning.
  • executor_worker.py: Approved code application.
Logic

codex_analysis

The "Brain" used by workers to understand tickets and formulate plans.

  • Rich prompt building from Jira manifests.
  • Codex CLI integration and logging.
  • Analysis artifact generation.
Logic

JiraStreamableMCP

Underlying service layer for robust Jira API interactions.

  • Recursive comment and attachment fetching.
  • Rate-limit aware streaming.
  • Detailed export manifest generation.
Ecosystem

Jira MCP Ecosystem

A collection of specialized MCP servers for different Jira interaction patterns.

  • JiraWebhookMCP: Manages ngrok tunnels and webhook registration.
  • JiraSSEMCP: Server-Sent Events implementation for real-time updates.
  • MCPJira: General-purpose Jira toolset for Codex.

OrchestratorMCP Worker Control Surface

OrchestratorMCP is no longer only an observation layer. It can now independently queue and dispatch any configured worker. This gives Codex two operating modes: workflow-level automation through events, or direct worker-level control for debugging and manual orchestration.

flowchart LR Codex[Codex / MCP Client] --> Cap[getWorkerCapabilities] Codex --> Enqueue[enqueueWorkerCommand] Codex --> Start[startWorker and worker-specific tools] Codex --> Dispatch[dispatchWorker / dispatchAllWorkers] Enqueue --> Store[exports/.orchestrator/commands] Start --> Store Dispatch --> Dispatcher[orchestrator/dispatcher.py] Store --> Dispatcher Dispatcher --> PM[process_manager.py] PM --> Export[export_jira_worker] PM --> Planner[planner_worker] PM --> Executor[executor_worker] PM --> PR[github_pr_worker] PM --> Review[reviewer_worker]
MCP Tool Capability When to use it
getWorkerCapabilities() Lists configured workers, enabled flags, instance counts, timeouts, max attempts, subprocess command/args, and payload hints. Use before dispatching to understand what workers exist and what input they expect.
enqueueWorkerCommand(workerType, workflowId, payload, allowOverlap) Creates a durable command directly under exports/.orchestrator/commands/<workerType>/pending/. Use when Codex should queue work but not start it immediately.
dispatchWorker(workerType, force, maxToStart) Claims pending commands for a worker type and starts subprocesses according to pool limits. Use to start queued work for one worker type.
dispatchAllWorkers(force) Attempts dispatch across every configured worker type. Use when Codex wants to drain all ready queues.
startWorker(workerType, workflowId, payload, force, allowOverlap) Generic one-shot command: enqueue any worker type and immediately dispatch one matching command. Use for advanced/manual worker control.
startExportJiraWorker(workflowId, issueUrl, force, allowOverlap) Directly starts workers.export_jira_worker. Use when a Jira issue should be fetched now, independent of webhook arrival.
startPlannerWorker(workflowId, issueDir, manifestPath, issueUrl, force, allowOverlap) Directly starts workers.planner_worker. Use when an exported issue should be analyzed by Codex now.
startExecutorWorker(workflowId, payload, force, allowOverlap) Directly starts workers.executor_worker. Currently a safe placeholder; enable only after coding automation is implemented.
startPullRequestWorker(workflowId, payload, force, allowOverlap) Directly starts workers.github_pr_worker. Currently a safe placeholder for future GitHub PR creation.
startReviewerWorker(workflowId, payload, force, allowOverlap) Directly starts workers.reviewer_worker. Currently a safe placeholder for future PR review automation.
processNextEvent() / processPendingEvents(limit) Processes pending events synchronously without requiring the long-running event loop. Use when Codex wants deterministic step-by-step control.
Direct worker starts still respect the per-workflow active-command guard by default. Set allowOverlap=true only for deliberate debugging. Set force=true only when intentionally dispatching a disabled worker, such as a future placeholder worker.

System Observation

The system is designed to be fully observable. Observation starts when the webhook server is registered with Jira, creating a feedback loop.

Durable Storage Layout

The orchestrator persists state in the exports/.orchestrator/ directory. This enables recovery after crashes or restarts.

.orchestrator/
├── events/                # Incoming event journal and pending queue
│   └── incoming/pending/ ...
├── commands/              # Per-type command queues
│   └── export_jira_issue/pending/ ...
├── workflows/             # Per-issue state and history
│   └── SCRUM-13/state.json
├── dedupe/                # Idempotency keys
└── logs/                  # Orchestrator system logs
