Sprinter Architecture

Sprinter is a local Jira and Confluence integration utility. The CLI performs the core export and create workflows. The optional MCP packages expose those workflows to local MCP clients while keeping large artifacts on disk.

Architecture Snapshot

The project is intentionally small. It has one CLI orchestration module, one HTTP integration module, one helper module, three MCP adapter packages, a webhook receiver package, and a Jira webhook management package.

Primary purpose

Export Jira issue data, linked Confluence content, attachments, logs, and manifests into local files.

Secondary purpose

Create Jira issues from JSON payloads and preserve the request and response artifacts.

Runtime model

Commands run locally, call Atlassian REST APIs over HTTPS, and write artifacts under storage.export_path.

Extension model

MCPJira runs over stdio. JiraStreamableMCP runs over Streamable HTTP. JiraSSEMCP runs over SSE. webhooks/ receives Jira pushes.

System Map

The CLI and MCP server are separate entrypoints, but they converge on the same fetcher and artifact-writing behavior.

CLI user
main.py parses export and create, loads config, builds fetchers, runs workflow functions, and writes manifests.
MCP clients
MCPJira/server.py serves stdio clients. JiraStreamableMCP/server.py serves Streamable HTTP clients. JiraSSEMCP/server.py serves SSE clients. Each package has its own service facade.
Jira webhooks
webhooks/server.py receives signed Jira events. webhooks/store.py deduplicates events and stores jobs. webhooks/worker.py exports the affected issue through the existing service facade.
Core modules
fetcher.py owns Jira and Confluence HTTP behavior. utils.py owns URL parsing, safe filenames, path collision handling, and file writes.
Atlassian APIs
Jira provides issue data, activity, remote links, attachments, and create-issue responses. Confluence provides page bodies, hierarchy, comments, and attachments.
Filesystem
exports/ is the durable contract for users and MCP clients. exports/.webhooks/ is the webhook event and job store.

Configuration and Dependency Boundaries

Configuration is shared by the CLI and MCP surfaces. Dependencies are split so the base CLI install stays minimal.

Base requirements

requirements.txt installs requests for HTTP and PyYAML for config parsing across the CLI, webhooks, and webhook API package.

MCP requirements

MCPJira/requirements.txt, JiraStreamableMCP/requirements.txt, and JiraSSEMCP/requirements.txt include the base requirements and add mcp[cli].

Auth strategy

Each service supports bearer tokens or basic auth. Secrets can be inline, but environment variables are the safer path.

Storage strategy

The filesystem is the persistence layer. Tool responses stay small and point to saved artifact paths, MCP resource URIs, or webhook job files.

jira:
  base_url: "https://your-domain.atlassian.net"
  auth:
    type: "basic"
    email_env: "ATLASSIAN_EMAIL"
    token_env: "ATLASSIAN_API_TOKEN"

confluence:
  base_url: "https://your-domain.atlassian.net/wiki"
  auth:
    type: "basic"
    email_env: "ATLASSIAN_EMAIL"
    token_env: "ATLASSIAN_API_TOKEN"

storage:
  export_path: "./exports"

Runtime Flows

These are the operational paths implemented by the current code, not aspirational architecture.

Export issue

  1. Parse a Jira issue key from the provided URL.
  2. Validate the URL host against jira.base_url.
  3. Fetch issue JSON, comments, worklogs, changelog, remote links, and attachments.
  4. Optionally download Jira attachment binaries.
  5. Resolve Confluence page IDs from same-host remote links.
  6. Export linked pages and configured descendants.
  7. Write export.log and export_manifest.json.

Create issue

  1. Load a JSON payload from a file in the CLI, or receive a payload dict through MCP.
  2. Validate the minimum create-issue shape before calling Jira.
  3. Send POST /rest/api/3/issue.
  4. Use the returned key or ID to choose the final artifact directory.
  5. Write request, response, source metadata, log, and manifest files.

MCP invocation

  1. For stdio, the client starts .venv/bin/python -m MCPJira.server.
  2. For HTTP, the operator starts .venv/bin/python -m JiraStreamableMCP.server.
  3. For SSE, the operator starts .venv/bin/python -m JiraSSEMCP.server.
  4. FastMCP exposes tool and resource schemas over the selected transport.
  5. The tool handler calls the package-specific service facade.
  6. The service runs the same core workflow functions used by the CLI.
  7. The client receives a compact summary and can read resources for larger JSON artifacts.

Webhook invocation

  1. Jira sends a signed issue-scoped event to /webhooks/jira.
  2. The webhook app verifies the local secret header or Jira X-Hub-Signature.
  3. The parser normalizes event type, issue key, project, actor, and raw payload.
  4. The filesystem store records the event and suppresses duplicate deliveries.
  5. Accepted events enqueue export jobs under exports/.webhooks/jobs/queued/.
  6. The worker exports the issue through the existing Sprinter service facade.
  7. Job state moves to success or failed, and the export manifest records webhook trigger metadata.

The manifest is the high-level run summary. The log is the chronological trace. The JSON files are the raw evidence for later review, replay, or debugging.

Webhook Surface

The webhook packages add a push-based path beside the CLI and MCP pull/invocation paths. They still reuse the same Jira export workflow.

Component Behavior
webhooks/server.py Starts a local stdlib HTTP server with /webhooks/jira, /health, /ready, and /jobs/{job_id}.
webhooks/app.py Coordinates verification, parsing, filtering, dedupe, record-only events, and job enqueueing.
webhooks/store.py Stores event records and job lifecycle files under exports/.webhooks/ using atomic JSON writes.
webhooks/setup.py Starts the webhook server and ngrok, discovers the public URL, registers the Jira admin webhook, and runs readiness checks.
webhookAPI/ Creates, lists, gets, deletes, registers, and refreshes Jira webhook registrations through Jira REST APIs.

MCP Surface

The MCP packages are adapters. They do not embed a model and do not create a second Atlassian API implementation.

Tool or resource Behavior
MCPJira Existing stdio package with tools named jira_export_issue, jira_create_issue, and artifact readers.
JiraStreamableMCP New Streamable HTTP package with tools named jira_stream_export_issue, jira_stream_create_issue, and HTTP server diagnostics.
JiraSSEMCP SSE package with tools named jira_sse_export_issue, jira_sse_create_issue, and SSE server diagnostics.
jira_export_issue(ticket_url) Exports a Jira issue and linked Confluence pages, then returns issue key, artifact paths, counts, and a manifest resource URI.
jira_create_issue(payload) Creates a Jira issue from a structured payload, then returns issue key, artifact paths, and a response resource URI.
jira_get_export_manifest(issue_key) Reads and returns an existing export manifest from disk.
jira_get_created_issue_response(issue_key) Reads and returns an existing created-issue response from disk.
sprinter://exports/{issue_key}/manifest Read-only JSON resource for an exported issue manifest.
sprinter://created/{issue_key}/response Read-only JSON resource for a created issue response.
jirastream://exports/{issue_key}/manifest Streamable HTTP package read-only JSON resource for an exported issue manifest.
jirastream://created/{issue_key}/response Streamable HTTP package read-only JSON resource for a created issue response.
jirasse://exports/{issue_key}/manifest SSE package read-only JSON resource for an exported issue manifest.
jirasse://created/{issue_key}/response SSE package read-only JSON resource for a created issue response.

Boundaries and Limits

These constraints are useful when extending the project or diagnosing missing data.

  • Confluence discovery starts from Jira remote links only.
  • Remote links are ignored unless their host matches confluence.base_url.
  • Legacy Confluence display links are resolved by search when a space key and title can be parsed.
  • Confluence page bodies are stored in storage format, not rendered HTML.
  • The workflows are synchronous and single-issue oriented.
  • All MCP packages expect local filesystem access to the same artifact directory as the CLI.
  • The HTTP MCP packages should stay bound to 127.0.0.1 unless protected by network controls.
  • The webhook server is local by default; Jira Cloud needs a public tunnel such as ngrok to deliver events to it.
  • Generated exports/ artifacts are machine-specific output and should not be treated as source documentation.
